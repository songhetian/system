# ADR-004: 系统安全基石优先于功能扩展

**状态**：提议
**日期**：2026-07-04
**decider**：架构审计

## 背景

系统完成了 12 模块的模型设计和约 55-60% 的代码骨架，但审计发现 5 个 P0 级安全/架构问题：

1. RBAC 权限定义了但零路由接入
2. 行级数据权限定义了但零服务调用
3. 字段脱敏实现了但零使用
4. 审批流与业务模块完全脱节
5. WebSocket 无心跳/内存泄漏/不可扩展

同类 HR 系统（北森、盖雅、钉钉人事）的共性是：安全管控是第一天就做好的基石，功能可以迭代。

## 决策

**暂停新功能开发，优先解决安全基石问题。** 具体优先级：

| 顺序 | 事项 | 预估工作量 |
|-----|------|-----------|
| 1 | 路由接入 RBAC 权限校验 | 2 天 |
| 2 | 行级数据权限落地（OWN + DEPARTMENT + ALL） | 3 天 |
| 3 | 字段脱敏接入服务层 | 1 天 |
| 4 | 审批流接入业务模块（请假/加班/报销） | 3 天 |
| 5 | 数据库 migrate 化 | 1 天 |
| — | **合计** | **约 10 个工作日** |

## 理由

- **合规要求**：HR 系统处理身份证、薪资等敏感数据，数据泄露 = 法律风险
- **业务正确性**：审批流不接入的话，请假/报销/加班创建即生效，没有管控
- **技术债务递增**：越晚接入权限，要改的代码越多，测试覆盖越难补
- **YAGNI 的反例**：安全不是"以后再说"的功能，是基础设施

## 后果

- 正面：系统从 Demo 级跨入生产可用级
- 负面：新功能开发暂停约 2 周
- 风险：数据权限接入可能影响现有测试（需要给测试用户分配正确权限）

## 替代方案

- **边做功能边补安全**：不推荐。容易忘、容易漏、测试不完整。
- **先做核心业务再补安全**：不推荐。HR 数据敏感，裸奔时间越长风险越大。
