## 1. 智能排班模块详细设计

### 1.1 排班模式定义（三种，逐步升级）

| 模式                   | 说明                                  | 难度    | 推荐阶段 | 功能范围      |
| -------------------- | ----------------------------------- | ----- | ---- | --------- |
| **模式 A：手动排班 + 冲突检测** | 人工拖拽排班，系统自动标红冲突（超时、连续夜班、人手不足）       | ⭐⭐    | 第一期  | 基础排班、冲突检测 |
| **模式 B：模板排班 + 人工微调** | 基于历史排班生成模板，自动套用，人工调整异常              | ⭐⭐⭐   | 第二期  | 模板管理、批量排班 |
| **模式 C：AI 优化排班**     | 基于 Prophet 预测人力需求 + Timefold 求解最优排班 | ⭐⭐⭐⭐⭐ | 第三期  | 智能优化、成本控制 |

**建议**：第一期先做模式 A，验证业务流程；第二期做模式 B，提升效率；第三期做模式 C，实现"智能"。

### 1.2 排班优化算法选型

#### 方案对比

| 方案                     | 优点                             | 缺点                      | 技术栈                | 推荐度   |
| ---------------------- | ------------------------------ | ----------------------- | ------------------ | ----- |

| **自研约束求解（后端 Node.js）** | 无额外服务，直接集成，技术栈统一               | 复杂约束难以优化，性能差，开发周期长      | Node.js + Piscina  | ⭐⭐    |

### 1.3 排班约束规则详细列表

#### 硬约束（必须满足，否则排班非法）

| 约束       | 说明                     | 校验规则                                      | 错误提示                                |
| -------- | ---------------------- | ----------------------------------------- | ----------------------------------- |
| 员工每日工时上限 | 员工每日工时 ≤ 法定上限（如 11 小时） | `dailyHours <= maxDailyHours`             | "员工 {name} 当日工时 {hours} 超出上限 {max}" |
| 员工每周工时上限 | 员工每周工时 ≤ 法定上限（如 60 小时） | `weeklyHours <= maxWeeklyHours`           | "员工 {name} 本周工时 {hours} 超出上限 {max}" |
| 连续工作休息约束 | 连续工作时间 ≤ 4 小时必须休息      | `workDuration <= 4h`                      | "员工 {name} 连续工作 {hours} 需休息"        |
| 夜班后休息约束  | 夜班后必须休息 ≥ 12 小时        | `restAfterNightShift >= 12h`              | "员工 {name} 夜班后休息不足 12 小时"           |
| 员工技能匹配   | 员工技能必须匹配岗位要求           | `employee.skills includes position.skill` | "员工 {name} 不具备 {skill} 技能"          |
| 最小在岗人数   | 每个时段在岗人数 ≥ 需求人数        | `staffCount >= requiredStaff`             | "{date} {hour}:00 在岗人数不足"           |

#### 软约束（尽量满足，影响排班质量评分）

| 约束          | 说明          | 优化目标      | 权重  |
| ----------- | ----------- | --------- | --- |
| 员工偏好时段      | 员工偏好时段优先安排  | 满足员工偏好    | 0.6 |
| 排班均衡        | 不让少数人加班过多   | 最小化工时方差   | 0.8 |
| 人力成本最小化     | 优先安排时薪低的员工  | 最小化总成本    | 0.7 |
| 员工连续工作天数最小化 | 避免员工连续工作过多天 | 最小化连续工作天数 | 0.5 |

## 2. 审批流规则引擎详细设计

### 2.1 条件分支支持类型

| 条件类型 | 示例                   | 数据结构                                                      | 前端配置界面                |
| ---- | -------------------- | --------------------------------------------------------- | --------------------- |
| 金额阈值 | 报销金额 > 5000 需总监审批    | `{ field: 'amount', operator: '>', value: 5000 }`         | 下拉选择字段 + 运算符 + 输入值    |
| 部门   | 研发部走技术总监，销售部走销售总监    | `{ field: 'department', operator: '==', value: '研发部' }`   | 下拉选择字段 + 运算符 + 下拉选择部门 |
| 岗位   | 主管级以上需 HR 审批         | `{ field: 'position_level', operator: '>=', value: 5 }`   | 下拉选择字段 + 运算符 + 输入值    |
| 请假天数 | 年假 > 3 天需总经理审批       | `{ field: 'leave_days', operator: '>', value: 3 }`        | 下拉选择字段 + 运算符 + 输入值    |
| 员工类型 | 正式员工走 A 流程，实习生走 B 流程 | `{ field: 'employee_type', operator: '==', value: '正式' }` | 下拉选择字段 + 运算符 + 下拉选择类型 |

| `>=`     | 大于等于 | 数字、日期  |

### 2.2 审批流向规则

#### 审批动作

| 动作             | 说明             | 前端界面                | 数据流                     |
| -------------- | -------------- | ------------------- | ----------------------- |
| **通过**         | 流转到下一个节点       | 按钮："通过"             | 更新 `currentNode` 为下一个节点 |
| **驳回（退回上一步）**  | 驳回到上一个审批节点     | 按钮："驳回" + 选择"退回上一步" | 更新 `currentNode` 为上一个节点 |
| **驳回（退回发起人）**  | 驳回到流程发起人       | 按钮："驳回" + 选择"退回发起人" | 更新 `currentNode` 为第一个节点 |
| **驳回（退回指定节点）** | 驳回到指定的审批节点     | 按钮："驳回" + 下拉选择节点    | 更新 `currentNode` 为指定节点  |
| **抄送**         | 仅通知，不算审批节点     | 按钮："抄送"             | 不阻塞流程，发送通知              |
| **加签**         | 在当前节点前插入一个审批节点 | 按钮："加签" + 选择审批人     | 动态修改流程实例的节点列表           |

#### 审批流数据存储设计

```prisma
model ApprovalFlow {

model ApprovalInstance {
  id          String   @id @default(uuid())

  businessType String   // 业务类型：leave/expense/schedule

  currentNode String   // 当前审批节点 ID
  status      String   @default("pending") // pending/approved/rejected/cancelled
  initiatorId String   // 发起人 ID
  history     Json     // 审批历史（谁审批的、意见、时间）

## 3. 文件安全分级详细设计

### 3.1 文件安全等级定义

| 等级     | 定义      | 预览要求          | 下载要求   | 示例文件      | 图标  |
| ------ | ------- | ------------- | ------ | --------- | --- |
| **公开** | 所有人可查看  | 直接预览          | 直接下载   | 公司公告、制度文件 | 🌐  |
| **内部** | 登录后可查看  | 直接预览          | 直接下载   | 培训课件、操作手册 | 🔒  |
| **敏感** | 需权限可查看  | 二级密码校验        | 二级密码校验 | 薪资报表、合同   | ⚠️  |
| **机密** | 仅限指定人查看 | 二级密码 + 操作审计日志 | 禁止下载   | 高管合同、财务报表 | 🔴  |

---

### 3.2 文件权限控制实现

#### 上传时

1. 用户选择文件
2. 选择文件等级（默认"内部"）
3. 选择可访问人员/角色（仅敏感/机密等级需要）
4. 上传到 MinIO
5. 保存文件元数据到数据库

#### 预览时

**流程**：

```
用户点击预览
  ↓
检查文件等级
  ↓
[公开/内部] → 直接预览
  ↓
[敏感/机密] → 弹出二级密码弹窗
  ↓
校验通过 → 生成临时 Token（有效期 10 分钟）
  ↓
打开预览弹窗（iframe 加载 kkFileView）
  ↓
[机密] → 记录操作审计日志
```

#### 下载时

**流程**：

```
用户点击下载
  ↓
检查文件等级
  ↓
[公开/内部] → 直接下载
  ↓
[敏感] → 弹出二级密码弹窗
  ↓
校验通过 → 生成下载链接（有效期 10 分钟）
  ↓
[机密] → 禁止下载，提示"机密文件仅可在线预览"
```

## 4. Excel导入导出详细设计补充

### 4.1 大文件定义

| 文件大小 | 行数                          | 处理方式        | 说明                 |
| ---- | --------------------------- | ----------- | ------------------ |
| 小文件  | ≤ 5MB / ≤ 2000 行            | 同步处理        | 直接解析，立即返回结果        |
| 中文件  | 5MB - 20MB / 2000 - 10000 行 | 异步队列        | BullMQ 异步处理，前端轮询进度 |
| 大文件  | > 20MB / > 10000 行          | 分片上传 + 异步队列 | 分片上传，合并后异步处理       |

  const interval = setInterval(async () => {
    const progress = await fetchTaskProgress(taskId);

    if (progress.status === 'completed') {
      clearInterval(interval);
      // 下载处理结果
      downloadResult(progress.resultFileId);
    } else if (progress.status === 'failed') {
      clearInterval(interval);
      // 显示错误
      showError(progress.errorMessage);
    } else {
      // 更新进度条
      updateProgressBar(progress.percent);
    }
  }, 2000);
};
```

## 7. 项目靠谱度评估

### 7.1 靠谱的地方 ✅

1. **技术栈选择正确**
   
   - Fastify（高性能）
   - Prisma（类型安全）
   - React Query（异步管理）
   - BullMQ（异步任务）
   - 这套栈能支撑中大型企业应用

2. **开源借鉴策略正确**
   
   - 只借鉴业务逻辑，不抄代码
   - 避免开源协议风险
   - UI 统一，用户体验好

3. **安全设计完善**
   
   - 二级密码
   - 字段脱敏
   - 操作审计
   - 这些在企业级系统里确实必要

4. **UI 规范明确**
   
   - Arco Design Pro 统一 UI
   - 商务蓝色主题
   - 禁用花哨动效
   - 保证系统中台质感

---

### 7.2 风险点 ⚠️

1. **开发工作量大**
   
   - 预估：3 人团队，12-18 个月
   - 如果团队只有 1-2 人，周期会很长
   - 建议：分三期做，先出 MVP

2. **排班优化算法难度大**
   
   - Timefold 需要学习
   - 对接调试需要时间
   - 建议：第一期先做手动排班 + 冲突检测

3. **HR 业务细节复杂**
   
   - 薪资计算（个税、社保、公积金）
   - 考勤规则（弹性工作制、倒班）
   - 合规约束（各地劳动法不同）
   - 建议：参考 ERPNext 的实现，或找 HR 专家咨询

4. **kkFileView 维护不活跃**
   
   - 最近更新：2023 年
   - 可能有安全漏洞
   - 备选方案：用 **OnlyOffice** 或 **Collabora Online**
