## 一、总体评价

**系统完成度约 55-60%**。基础骨架和核心域模型搭建扎实，但关键业务逻辑、安全控制、运维体系大量缺失。当前状态：**Demo 级可用，生产级不足**。

好的方面：
- Prisma schema 设计全面，12 模块 40+ 表，关联关系完整
- TDD 执行到位，262 个测试覆盖核心服务
- 技术选型合理（Fastify + Prisma + React Query + Zod）
- 前后端共享 Zod schema，类型一致性有保障

核心问题：
- **权限系统完全是空壳** — `requirePermission` 定义了但零调用
- **数据权限（行级隔离）完全未落地** — 普通员工能看全公司数据
- **字段脱敏未接入服务层** — 身份证、薪资、手机号裸奔
- **审批流与业务模块脱节** — 请假/加班/报销创建后不启动审批流
- **大量关键业务功能缺失** — 见第三章

## 二、安全与架构问题（P0 - 必须修复）

### P0-1 权限系统未接入任何路由

**现状**：`requirePermission()` 函数在 [auth.ts](file:///e:/shop/system/server/src/middleware/auth.ts#L128-L146) 中定义，但 20+ 路由文件中 **零调用**。

**影响**：任何登录用户可以调用任何接口 —— 普通员工能删部门、改薪资、删用户。

**同类项目参考**：
- 钉钉/飞书人事系统：RBAC + 数据权限双重控制
- 北森/盖雅：菜单权限 + 按钮权限 + 数据范围三级管控

**修复方案**：
1. 为每个路由挂上对应的 permission code（如 `employee:read`, `employee:write`）
2. 路由级别统一通过 preHandler hook 注入
3. 前端路由守卫同步校验

---

### P0-2 行级数据权限完全未生效

**现状**：`buildRowAccessFilter()` 在 [row-access.ts](file:///e:/shop/system/server/src/utils/row-access.ts) 中定义，但 **零服务调用**。注释写着 "Phase 1 简化实现"，但 Phase 1 都没接。

**影响**：
- 员工 A 可以看到员工 B 的薪资条
- 员工可以查询全公司的考勤/请假/报销数据
- 部门主管身份形同虚设

**修复方案**：
1. 所有列表查询 service 必须调用 `buildRowAccessFilter` 注入过滤条件
2. 实现 `DEPARTMENT` / `DEPARTMENT_AND_CHILDREN` 两种范围的递归过滤
3. 单条查询（getById）也要校验数据权限，防止越权

---

### P0-3 字段脱敏未接入输出

**现状**：`maskPhone` / `maskIdCard` / `maskSalary` / `maskEmail` 在 [mask.ts](file:///e:/shop/system/server/src/utils/mask.ts) 中实现，但 **零服务调用**。

**影响**：
- 员工列表接口返回所有人的身份证号、手机号、邮箱明文
- 薪资接口无权限校验也无脱敏

**修复方案**：
1. 列表查询默认脱敏，详情查询根据权限决定是否明文
2. 薪资字段永远脱敏，二级密码验证通过后临时明文
3. 建立统一的 response interceptor / transformer 模式

---

### P0-4 审批流与业务模块脱节

**现状**：
- `LeaveRequest` 有 `workflowInstanceId` 字段，但创建请假时**不启动审批流**
- `OvertimeRequest` 同上
- `ExpenseClaim` 同上
- 审批通过后也没有回调更新业务单据状态

**影响**：审批流模块是孤岛，业务模块裸奔，请假单创建了就生效，没人审批也能扣额度。

**修复方案**：
1. 请假/加班/报销创建时自动启动对应审批流
2. 审批通过/驳回回调更新业务单据状态
3. 审批通过后才执行扣额度、计入考勤等业务逻辑

---

### P0-5 WebSocket 连接无心跳检测 + 内存泄漏风险

**现状**：[websocket.ts](file:///e:/shop/system/server/src/plugins/websocket.ts)
- 连接存在内存 Map 中，进程重启全丢
- 没有心跳超时检测，死连接不清理
- 单机内存存储，多实例部署不通

**影响**：连接泄漏、消息丢失、水平扩展不可能。

**修复方案**：
1. 加 ping/pong 心跳超时（30s 无响应断开）
2. 用 Redis Pub/Sub 做跨实例消息广播
3. 连接状态存储到 Redis，支持多实例

## 四、运维与可靠性问题（P1 - 生产必备）

### 4.1 无统一日志体系

**现状**：
- 没有结构化日志（JSON log）
- 没有 request-id 链路追踪
- 错误日志散落各处，没有集中收集
- 没有日志分级（debug/info/warn/error）

**修复方案**：接入 `pino`（Fastify 生态原生支持），统一 request-id，关键操作埋点。

### 4.2 无健康检查接口

**现状**：没有 `/health` 接口，K8s/Docker 健康检查没法做。

### 4.3 无应用监控/指标

**现状**：
- 没有 Prometheus metrics 接口
- 没有 API 响应时间统计
- 没有错误率监控
- 没有慢查询监控

### 4.4 无数据库迁移管理

**现状**：`prisma/migrations/` 目录下只有 `migration_lock.toml`，没有真正的 migration 文件。都是用 `db push` 直接推的。

**风险**：生产环境 schema 变更无法回滚、无法追踪、无法团队协作。

### 4.5 无备份策略

**现状**：没有数据库定期备份脚本、没有 MinIO 文件备份方案。

### 4.6 配置管理混乱

**现状**：`.env` 文件散落在各处（根目录、server/prisma/），没有统一的配置加载和校验。

## 五、代码质量与架构问题（P2 - 技术债）

### 5.1 前端页面大量功能缺失

后端 12 个模块都有路由和 service，但前端只实现了 7-8 个页面，且大多是"能看不能用"的列表页。

未实现/半成品的前端页面：
- 员工档案详情（合同、证件、变动记录）
- 薪资模块（计算、发放、工资条查看）
- 报销申请/审批
- 培训管理
- 角色权限管理
- 数据权限配置

### 5.2 前端没有路由级权限控制

**现状**：所有菜单都显示，所有页面都能进，只是调接口会被后端挡（但后端目前也没挡）。

### 5.3 前端没有全局错误处理

**现状**：每个页面自己 try/catch，没有统一的错误边界（Error Boundary）和全局错误提示。

### 5.4 Excel 导入导出只有骨架

**现状**：`ExcelTask` 模型 + `excel.ts` 路由，但实际导入逻辑是空的或只有员工模块。

### 5.5 消息通知机制不完整

**现状**：
- Message 模型有，但业务事件（审批通过、请假驳回等）不发消息
- 只有 WebSocket 推送，没有站内信列表的未读红点以外的通知
- 没有邮件/短信通知渠道

## 六、优先级排序与建议路线图

### 第一阶段：安全底线（1-2 周）— **不做不能上线**

1. **P0-1** 路由接入权限校验（RBAC）
2. **P0-2** 行级数据权限落地（至少 OWN + DEPARTMENT + ALL）
3. **P0-3** 字段脱敏接入服务层（身份证、手机号、薪资）
4. **P0-4** 审批流接入业务模块（请假/加班/报销走审批）
5. 数据库迁移（从 db push 切到 migrate）

### 第二阶段：核心业务闭环（3-4 周）— **能用**

6. 薪资计算引擎（基本工资 + 考勤扣款 + 加班费 + 社保 + 个税）
7. 考勤完整闭环（打卡导入 → 自动判定迟到早退 → 生成台账 → 联动薪资）
8. 排班发布流程 + 排班与请假联动
9. 年假自动计算 + 额度结转
10. 报销完整流程（申请→审批→付款登记）

### 第三阶段：体验与效率（2-3 周）— **好用**

11. 审批流高级功能（会签、条件分支、转签）
12. Excel 全模块导入导出
13. 消息中心完善（站内信 + 待办 + 邮件通知）
14. 前端路由权限 + 按钮级权限
15. 入职/转正/离职/调动流程

### 第四阶段：运维与规模化（持续）— **稳**

16. 结构化日志 + 链路追踪
17. 监控指标 + 告警
18. 健康检查 + 备份策略
19. WebSocket 水平扩展（Redis Pub/Sub）
20. 审批流超时定时任务 + 其他定时任务
